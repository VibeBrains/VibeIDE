# Сторонний код и провенанс — флажок ДО сборки/релиза

← [Knowledge Index](../README.md)

**Контекст.** В релиз v0.16.0 ушло встроенное расширение `vibe-keybindings`, представлявшее собой дословный вендоринг чужого MIT-проекта (kasecato/vscode-intellij-idea-keybindings) с его копирайтом. Проблема всплыла уже ПОСЛЕ публикации релиза (по ченджлогу), релиз пришлось удалять и переавторивать keymap. Пользователь: «впредь предупреждай о такой лаже».

**Суть (правило поведения агента).**
- Любой код/данные, взятые из стороннего репозитория (вендоринг, порт, «по образцу»), — это **провенанс-риск**, его нужно поднять ЯВНО и ДО того, как он попадёт в `git commit` → билд → релиз. Не молча тащить и не «разберёмся потом».
- «Это же опен-сорс / просто данные / просто сочетания клавиш» — **не** отменяет лицензию апстрима. Open source ≠ public domain. MIT/BSD/Apache требуют сохранять копирайт-уведомление при копировании существенной части.
- «Сделаем как у них, но наше» при дословной копии файла — это производная работа, а не независимое авторство. «Наше» = переавторить инженерную часть (у keymap: `when`-клаузы, выбор сопоставлений, кураторскую подборку), а функциональные факты (схема горячих клавиш JetBrains, ID команд VS Code) совпадать могут и это не нарушение.
- Самый плохой исход — оставить копию, но снять атрибуцию: это прямое нарушение лицензии. Либо честная атрибуция, либо честное переавторивание.

**Применение.**
1. Перед бандлингом стороннего артефакта — назвать пользователю: источник, лицензию, что именно копируется (весь файл / подмножество / только идея) и вытекающие обязательства. Дождаться решения.
2. Никогда не доводить такой артефакт до `release-windows.ps1` без явного «ок» по лицензии.
3. Если решение «сделать своим» — переавторить, а не переименовать; убрать `repository.url`/копирайт апстрима, задокументировать наш провенанс (на основе какой публичной спецификации построено).
4. Связано: [vibe-keybindings.md](vibe-keybindings.md) (история этого конкретного кейса и текущая модель владения keymap).
