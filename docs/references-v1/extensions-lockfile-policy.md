# Extensions lockfile policy (VibeIDE vs microsoft/vscode sync)

**Цель:** один источник правды для npm lockfiles каталога `extensions/`, чтобы upstream-merge не оставлял сотни неотслеживаемых `package-lock.json` и не ломал контрибьюторов ложными конфликтами.

## Решение по умолчанию (текущая практика OSS VS Code)

- **Корневой** `package-lock.json` и скрипты `npm install` подтягивают зависимости встроенных расширений согласно корневому дереву.
- Отдельные `extensions/*/package-lock.json` появляются **только** когда расширение реально ведёт свой npm-пакет и это осознанно закреплено в репозитории (или когда скрипт сборки это генерирует в CI).

## Правила для VibeIDE при merge upstream

1. Не коммитить массово новые `package-lock.json` без проверки `product.json` → список builtin extensions и корневых скриптов install.
2. При конфликте lockfile: предпочесть **перегенерацию** из поддерживаемого шага CI (`npm ci` / документированный install), а не ручное склеивание тысяч строк.
3. CI должен иметь **одну** команду воспроизведения зависимостей (`README` / `HOW_TO_CONTRIBUTE.md`); расхождение «локально ок / CI красный» трактовать как нарушение этого документа.

## Будущая автоматизация (backlog)

- Явная проверка в workflow: «нет неучтённых lockfiles под `extensions/`» или наоборот whitelist каталогов — только после фиксации списка расширений с отдельным npm-пакетом.
